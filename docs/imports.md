# Imports Pipeline Foundation

This document defines the import domain and orchestration flow for Investment Control Center.

## Goals
- Deterministic parsing for broker and custom reports.
- End-to-end traceability from normalized rows back to batch + source file.
- Support transactional imports now and derived snapshots later.
- Ready for UI upload, watched directory volume, and FTP/PGP integration.

## Domain model
- `import_batch`: lifecycle container for a single ingestion operation.
- `import_file`: source artifact metadata (checksum, storage URI, encryption flag, format hints).
- `import_error`: batch/file errors (validation, parser detection, orchestration failures).
- `import_row_error`: row-level diagnostics generated by parser.
- `source_mapping_metadata`: versioned schema mapping definitions per source/object.

## Pipeline stages
1. **Upload intake** (`POST /api/v1/imports/intake`)
   - validates idempotency by SHA-256 checksum
   - stores file bytes in `import_storage`
   - creates `import_batch` + `import_file`
2. **Validation**
   - byte payload non-empty
   - UTF-8 decode check
   - header check
3. **Format detection**
   - CSV header extraction
   - parser registry match against source + headers
4. **Parsing orchestration**
   - parser produces normalized contracts with provenance metadata
   - writes transactional entities (currently trades)
5. **Error capture + audit logging**
   - capture batch errors in `import_error`
   - row issues in `import_row_error`
   - audit events in `audit_logs`

## Async execution
- Celery task `parse_import_batch_async` resolves stored file path and runs parsing.

## Provenance contract
Every normalized record includes:
- `source_file`
- `source_row` (if row-backed)
- `import_batch_id`
- `parser_version`

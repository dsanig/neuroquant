# Architecture - Investment Control Center

## Design goals
- Deterministic, auditable financial calculations.
- Clear separation between transactional ledger records and derived analytics snapshots.
- Scalable from one account/firm to multi-account and multi-user.

## System overview
- **Frontend**: Next.js App Router, TypeScript, server-rendered authenticated app shell.
- **Backend**: FastAPI with layered architecture (`api -> services -> repositories -> models`).
- **Data**: PostgreSQL normalized relational model with immutable audit fields on each entity.
- **Async**: Celery workers on Redis for ingestion, recomputation, and backfills.
- **Edge**: Nginx reverse proxy for internal TLS termination and route fanout.

## API versioning
All endpoints start at `/api/v1` to protect future compatibility.

## Auth architecture
- Internal first-party auth using hashed password credentials.
- Signed JWT access token generated by API.
- RBAC represented by `roles` and `user_roles` tables.
- Future: short-lived access + rotating refresh tokens + SSO provider.

## Core business logic
### Roll detection
A roll is detected if and only if:
1. buy and sell occur at exact same second,
2. same underlying,
3. matching absolute quantity.

Strikes may differ. Matching trades receive a deterministic `roll_group_id`.

### Margin formulas
- CSP capital at risk: `strike * multiplier * contracts`.
- Covered call notional: `current_price * multiplier * contracts`.

Both broker-reported and app-calculated metrics are stored via `margin_metrics.source`.

## Data model and indexing strategy
- High-frequency query indexes on:
  - `trades.executed_at`, `trades.symbol`, `trades.roll_group_id`
  - `margin_metrics.measured_at`, `margin_metrics.strategy_id`
  - `risk_metrics.measured_at`, `risk_metrics.metric_name`
  - `audit_logs.event_type`, `audit_logs.entity_type`, `audit_logs.entity_id`
- Unique external trade key: `(broker_account_id, external_trade_id)`.
- Soft deletion support (`is_deleted`) without destructive loss.

## Reliability choices
- DB readiness uses retry/backoff (`tenacity`) rather than startup sleep loops.
- Celery tasks configured with retries and late acks.
- Healthchecks for db, redis, backend, frontend in compose.

## Operational deployment
- `compose.yaml`: local baseline stack.
- `compose.production.yaml`: hardened runtime and optional monitoring profile.
- Persistent volumes for PostgreSQL, Redis, and uploads.

## Extensibility path
- Multi-account by linking strategies/positions/trades to `broker_accounts`.
- Strategy segmentation already first-class (`strategy_id` in positions/trades/margin).
- Additional broker import adapters can be added to tasks/services without router churn.
